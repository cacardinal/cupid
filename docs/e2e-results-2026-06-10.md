## Five real engine bugs found and FIXED during this run
(all verified fixed live; unit tests still 101/101)

1. **Nightly matching crashed on every pair** — `createMatchRecord` was passed
   `userAccepted: undefined`, which Firestore rejects. The scheduled job has
   never successfully created a match in production shape. Fixed by omitting
   the fields (`scheduler/jobs.ts`).
2. **Match proposals leaked raw JSON into SMS** — `max_tokens: 256` truncated
   Claude's `<profile_update>` block mid-JSON; the closing tag never arrived so
   the strip regex didn't match and users would receive partial JSON.
   Fixed: `parseClaudeResponse` now strips unterminated blocks (`services/claude.ts`).
3. **`detectIntent("yes")` returned "ambiguous"** — Claude over-hedged on the
   bare word with the old prompt, so accepting a match with "yes" got the
   clarification loop. Fixed: keyword fast-path via existing `detectYesNoIntent`
   + a contextualized prompt for genuinely unclear replies.
4. **Video link sent to only one side** — `createVideoRoom` updated both match
   records but only texted the user who accepted second. Fixed: looks up the
   other phone and sends both simultaneously (verified live during the demo
   video recording — both phones showed links).
5. **Contact-exchange deadlock** — `getActiveMatchForUser` filters out
   `contact_shared`, so once the first user consented, the second user's "yes"
   couldn't find their record and BOTH sides got "waiting to hear back" forever.
   Fixed: new status-independent `getMatchBetween()` + composite index added
   to `firestore.indexes.json`. Verified: second consenter now receives the
   exchange message.
